Radar SDK Documentation
Infineon software Radar SDK for the development of applications with Infineon radar devices.
Introduction to Radar SDK

Overview

The Radar SDK allows to evaluate the Infineon 60GHz radar sensors. It provides access to radar sensors from C/C++, Python, and Matlab. In addition, the Radar SDK contains low-level radar algorithms like Range-Doppler or Range-Spectrum, and high-level algorithms like presence sensing or segmentation.

The Radar SDK follows following design goals:

  • Portable: The software is written in standard C/C++, it compiles and runs on several platforms and operating systems. Most of the source code is available and can be integrated into existing projects or ported to new platforms or operating systems.
  • Modular: The Radar SDK consists of different modules grouping source files with related functionality. The modular structure allows to use parts of the Radar SDK. For instance, it is possible to use the device access layer (Device Control) without the radar algorithm module (Radar (ifxRadar)).
  • Simple: Wrappers for Python and Matlab are available. Examples illustrating the use of the Radar SDK and the Infineon radar chips are available for C, Python, and Matlab.

The SDK can be used in different scenarios:

  • Running the example applications to evaluate for instance the presence sensing or segmentation algorithm.
  • Development of a radar algorithm. This is typically done using the Python or Matlab wrapper.
  • Running experiments to find a suitable device configuration for a specific use-case.
  • Exporting a device configuration in form of a register list that can be used directly in a firmware development project.

To get started with RDK it is recommended:

  • Build the provided example applications and run them.
  • Python users can get started by reading the Python wrapper usage.
  • Matlab users can get started by reading the MATLAB wrapper usage.
  • Users who want to write an application in C or C++ should have a look at the example application provided as "raw_data" which illustrates how to fetch time-domain data from the radar sensor.

The C library

Modules

The Radar SDK consists of different modules that group source files with related functionality. All header and source files of a module are in a separate directory, for example ifxBase or ifxAlgo.

The Radar SDK contains these modules:

  • ifxAlgo: The algo module provides algorithms that are not radar-specific. Examples for such algorithms are FFT, windowing, or signal processing functionality.
  • ifxBase: The base module includes functionality that is used by all other modules. For example, ifxBase includes error handling, memory management, definitions for vectors and matrices, and math functions.
  • ifxDevice: The DeviceControl module provides access to the radar sensor. Among others it allows to connect to a radar sensor, to apply a device configuration, and fetch time-domain radar data from the sensor.
  • ifxRadar: The radar module contains radar-specific algorithms like Range-Doppler map or a presence detection.
  • ifxRadarSegmentation: This module contains the radar segmentation algorithm.

Conventions

  • All public APIs are compatible to C (C99).
  • Each function starts with the prefix ifx_.
  • Math functions that act on real numbers end with the suffix _r (similar for types).
  • Math functions that act on complex numbers end with the suffix _c (similar for types).
  • Mnemonics are typically shortened, e.g. sub instead of subtract or add instead of addition.
  • Each module contains one header file named like the module that includes all header files of the modules. For example, to include all header files of the module ifxAlgo it is sufficient to include ifxAlgo/Algo.h.

Object Oriented Programming and Memory Management

All public APIs of the Radar SDK can be used from C. The APIs mimick the object oriented programming (OOP) paradigm. Functions that end with the suffix _create play the role of constructors and return a pointer ("an object"). This pointer is then passed as the first argument to other functions of this "class". Functions that end with the suffix _destroy play the role of destructors and free the allocated ressources.

For example, the function ifx_mat_create_r creates a new real matrix and returns a pointer to a ifx_Matrix_R_t structure. This pointer can be passed as the first argument to the function ifx_mat_transpose_r to transpose the matrix. To free the matrix the function ifx_mat_destroy_r must be called.

Memory is typically allocated by a create function (a function with suffix _create) and freed by calling the corresponding destroy function (ending with the suffix _destroy). A few functions return a pointer that must be freed by the caller by passing it to ifx_mem_free. In such cases this behavior is documented in the API documentation.

Error Handling

The error handling works similar to the error handling using errno in C. Each function of the Radar SDK might set an error which can be checked using the function ifx_error_get. The function ifx_error_get returns an error code of type ifx_Error_t. To convert the error code into a human-readable string the function ifx_error_to_string can be used. This example illustrates how to check for an error in the case of matrix inversion:

// invert Matrix A
ifx_la_invert_r(A, Ainv);
ifx_Error_t error = ifx_error_get();
if(error != IFX_OK)
{
fprintf(stderr, "Invering matrix failed: %s (%d)\n", ifx_error_to_string(error), error);
...
}

The error code is sticky and must be explicitly cleared by calling ifx_error_clear. Radar SDK functions gracefully check input parameters such that passing NULL pointers or other invalid input values will not lead to a crash. This allows users to check for an error only after several Radar SDK function calls.

The error code is thread-local. This means that ifx_error_get returns the last error that occured in the current thread. Similarily, ifx_error_clear clears the error in the current thread.

Sometimes it is helpful to immidiately get a notification once an error occurs. This can be achieved by registering an error callback function using ifx_error_set_callback:

void error_callback(const char* filename, const char* functionname, int line, ifx_Error_t error)
{
fprintf(stderr, "An error occured in file %s, line %d, function %s: %s\n", filename, line, functionname, ifx_error_to_string(error));
}
int main()
{
// register error callback
ifx_error_set_callback(error_callback);
}

Thread safety

If possible use an object only in the same thread in which the object was created by calling a creator function (a function with suffix _create). If this is not possible, you must protect the object from simultaneous access using locks.

Opening and closing radar device is not thread-safe. The corresponding functions like ifx_device_create and ifx_device_destroy must not be called simultaneously from different threads.

A radar device must not be accessed from different threads at the same time. In other words, do not access the same device handle (ifx_Device_Handle_t) simultaneously. It is recommended to use the same device handle only from one thread. If you must use the same device handle from different threads, you must ensure that the device handle is not accessed simultaneously using a locking mechanism.

Logging

The Radar SDK will print debugging information to the console if the library has been built in Debug mode. If the library has been built in Release mode no such logging information will be printed.

Exporting a Device Configuration for a Firmware project

Often the Radar SDK is used for evaluation, finding a suitable device configuration for the radar sensor, or algorithm development. As a next step users might want to export the device configuration set by Device Control to a firmware project. This is possible using the function ifx_device_register_list_string which returns the register values of the radar sensor corresponding to the current configuration. In a firmware project these register values can be send to the radar sensor to set the exact same radar configuration.

The xsensix-bgt60trxx-driver is a platform-independent driver for Avian radar sensors. This driver allows to configure the radar sensor using the register list exported by ifx_device_register_list_string. In addition, the driver allows to read and write registers of the radar sensor, perform a soft reset, get and set the threshold for the FIFO interrupt, and to fetch time-domain data.

For more details see the documentation of ifx_device_register_list_string. Python and Matlab offer similar functions.

Device Access

Users not interested in the architecture of device access who only want to access the radar sensor may skip this section and should either refer to Device Control when using C/C++, to the Python wrapper usage or to the MATLAB wrapper usage. This section explains the different components involved in the device access and how they interact.

Figure 1 depicts the different components involved in the device access and their interaction between them. The components libAvian, strata/stratula, and DeviceControland, are explained in the following subsections.

Figure 1 Overview of the interaction of the different components.

libAvian

Avian is the marketing name of the BGT radar devices. The library libAvian is a utility library which contains utility classes and functions for controlling Avian devices. The key functionality is a class that translates configuration parameters (e.g., RF frequency range, ADC sample rate, number of samples, etc.) into register values that can be programmed into Avian devices. Additionally, libAvian allows to measure device temperature and emitted TX power. Some functions for hardware testing are also provided.

Strata/Stratula

Strata/stratula is a framework for developing software for multiple platforms. It allows to access interfaces on the firmware like SPI or I2C from the host, and it gives access to component interfaces. A component interfaces for example might allow to read or write registers inside the radar sensor.

Strata/stratula allows to communicate with the radar baseboard. In particular, it provides functionality to read and write registers the BGT radar devices, and to fetch radar data from BGT radar devices.

Strata is the C++ library used in the SDK host side code, while stratula is the C-version counterpart used in the firmware.

DeviceControl

Device Control (module ifxDevice) is part of the Radar SDK. It combines strata/stratula with libAvian and allows a user-friendly interface to the radar sensors. DeviceControl allows to connect to radar sensors, to configure radar sensor, and to fetch radar data. The interface is available from C/C++, wrappers for Python and Matlab are also available.

Interaction between DeviceControl, Strata/Stratula, and libAvian

  • DeviceControl uses libAvian to convert a high-level configuration into register values of the radar sensor.
  • DeviceControl uses Strata to communicate with the radar baseboard. In particular, Strata is used to fetch the time-domain radar data from the radar sensor.
  • libAvian uses Strata to access the registers of the radar sensor, to read the temperature from the radar sensor, and to retrieve the current RF transmission power.

Combining DeviceControl and libAvian

For most users the functionality provided by Device Control will be sufficient. In this case it is not recommended to interact directly with libAvian. In rare situations Device Control might not offer access to a configuration parameter that the user needs to set. In this case it is possible to get a handle to the libAvian driver using ifx_device_get_avian_driver and set the desired parameters directly via libAvian. Please note that it is only possible to access libAvian from C++.

This example illustrates how to set the cutoff frequency of the high-pass filter to 20kHz using libAvian:

#include "ifxDevice/DeviceControlHelper.hpp"
#include <ifxAvian_Driver.hpp>
// ...
// Initialize the device configuration structure
ifx_Device_Config_t config = { ... };
// Open device
ifx_Device_Handle_t* device_handle = ifx_device_create();
// Apply the device settings based on the device configuration structure.
ifx_device_set_config(device_handle, &device_config);
// Get avian driver
auto avian_driver = ifx_device_get_avian_driver(device_handle);
// Get the current baseband configuration and set hpCutoff to 20kHz for all
// (potentially 4) antennas
Infineon::Avian::Baseband_Configuration baseband_config = {};
avian_driver->get_baseband_configuration(&baseband_config);
baseband_config.hp_cutoff_1 = Infineon::Avian::RADAR_BGT6X_HP_CUTOFF_20KHZ;
baseband_config.hp_cutoff_2 = Infineon::Avian::RADAR_BGT6X_HP_CUTOFF_20KHZ;
baseband_config.hp_cutoff_3 = Infineon::Avian::RADAR_BGT6X_HP_CUTOFF_20KHZ;
baseband_config.hp_cutoff_4 = Infineon::Avian::RADAR_BGT6X_HP_CUTOFF_20KHZ;
auto ret = avian_driver->set_baseband_configuration(&baseband_config);
if (ret != Infineon::Avian::Driver::Error::OK)
{
// error handling
}

Important note: The user should be aware that ifx_device_set_config will overwrite the current libAvian configuration. That is why it is important first to set the device configuration using ifx_device_set_config and then in a second step adjust the additional configuration parameters using libAvian.
